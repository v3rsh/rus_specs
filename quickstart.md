# Руководство по быстрому старту

Это руководство поможет вам начать работу с Spec-Driven Development, используя Spec Kit.

> [!NOTE]
> Все скрипты автоматизации теперь предоставляют варианты как для Bash (`.sh`), так и для PowerShell (`.ps1`). CLI `specify` автоматически выбирает тип скрипта на основе ОС, если вы не передадите `--script sh|ps`.

## Процесс из 6 шагов

> [!TIP]
> **Осознание контекста**: Команды Spec Kit автоматически определяют активную функцию на основе вашей текущей Git ветки (например, `001-feature-name`). Для переключения между различными спецификациями просто переключите Git ветки.

### Шаг 1: Установка Specify

**В вашем терминале** выполните команду CLI `specify` для инициализации вашего проекта:

```bash
# Создать новую директорию проекта
uvx --from git+https://github.com/github/spec-kit.git specify init <PROJECT_NAME>

# ИЛИ инициализировать в текущей директории
uvx --from git+https://github.com/github/spec-kit.git specify init .
```

Явно выбрать тип скрипта (опционально):

```bash
uvx --from git+https://github.com/github/spec-kit.git specify init <PROJECT_NAME> --script ps  # Принудительно PowerShell
uvx --from git+https://github.com/github/spec-kit.git specify init <PROJECT_NAME> --script sh  # Принудительно POSIX shell
```

### Шаг 2: Определите вашу конституцию

**В интерфейсе чата вашего AI-агента** используйте команду `/speckit.constitution` для установки основных правил и принципов вашего проекта. Вы должны предоставить конкретные принципы вашего проекта в качестве аргументов.

```markdown
/speckit.constitution Этот проект следует подходу "Library-First". Все функции должны быть реализованы сначала как автономные библиотеки. Мы строго используем TDD. Мы предпочитаем паттерны функционального программирования.
```

### Шаг 3: Создайте спецификацию

**В чате** используйте команду `/speckit.specify` для описания того, что вы хотите построить. Сосредоточьтесь на **что** и **почему**, а не на технологическом стеке.

```markdown
/speckit.specify Построить приложение, которое поможет мне организовать мои фотографии в отдельные фотоальбомы. Альбомы группируются по дате и могут быть реорганизованы путем перетаскивания на главной странице. Альбомы никогда не находятся в других вложенных альбомах. В каждом альбоме фотографии предварительно просматриваются в интерфейсе, похожем на плитку.
```

### Шаг 4: Уточните спецификацию

**В чате** используйте команду `/speckit.clarify` для выявления и разрешения неоднозначностей в вашей спецификации. Вы можете предоставить конкретные области фокуса в качестве аргументов.

```bash
/speckit.clarify Сосредоточиться на требованиях безопасности и производительности.
```

### Шаг 5: Создайте технический план реализации

**В чате** используйте команду `/speckit.plan` для предоставления вашего технологического стека и архитектурных решений.

```markdown
/speckit.plan Приложение использует Vite с минимальным количеством библиотек. Используйте vanilla HTML, CSS и JavaScript насколько это возможно. Изображения никуда не загружаются, а метаданные хранятся в локальной базе данных SQLite.
```

### Шаг 6: Разбейте на задачи и реализуйте

**В чате** используйте команду `/speckit.tasks` для создания списка действенных задач.

```markdown
/speckit.tasks
```

Опционально, проверьте план с помощью `/speckit.analyze`:

```markdown
/speckit.analyze
```

Затем используйте команду `/speckit.implement` для выполнения плана.

```markdown
/speckit.implement
```

## Подробный пример: Построение Taskify

Вот полный пример построения платформы продуктивности команды:

### Шаг 1: Определите конституцию

Инициализируйте конституцию проекта для установки основных правил:

```markdown
/speckit.constitution Taskify — это приложение "Security-First". Все пользовательские вводы должны быть проверены. Мы используем микросервисную архитектуру. Код должен быть полностью документирован.
```

### Шаг 2: Определите требования с помощью `/speckit.specify`

```text
Разработать Taskify, платформу продуктивности команды. Она должна позволять пользователям создавать проекты, добавлять членов команды,
назначать задачи, комментировать и перемещать задачи между досками в стиле Kanban. В этой начальной фазе для этой функции,
назовем её "Create Taskify", давайте иметь нескольких пользователей, но пользователи будут объявлены заранее, предопределены.
Я хочу пять пользователей в двух разных категориях, один менеджер продукта и четыре инженера. Давайте создадим три
разных примера проекта. Давайте иметь стандартные колонки Kanban для статуса каждой задачи, такие как "To Do",
"In Progress", "In Review" и "Done". Не будет входа в систему для этого приложения, так как это просто самая
первая тестовая вещь, чтобы убедиться, что наши основные функции настроены.
```

### Шаг 3: Уточните спецификацию

Используйте команду `/speckit.clarify` для интерактивного разрешения любых неоднозначностей в вашей спецификации. Вы также можете предоставить конкретные детали, которые хотите убедиться, что включены.

```bash
/speckit.clarify Я хочу уточнить детали карточки задачи. Для каждой задачи в UI для карточки задачи вы должны иметь возможность изменить текущий статус задачи между различными колонками на рабочей доске Kanban. Вы должны иметь возможность оставить неограниченное количество комментариев для конкретной карточки. Вы должны иметь возможность, из этой карточки задачи, назначить одного из действительных пользователей.
```

Вы можете продолжить уточнение спецификации с помощью `/speckit.clarify`:

```bash
/speckit.clarify Когда вы впервые запускаете Taskify, он покажет вам список из пяти пользователей для выбора. Пароль не потребуется. Когда вы нажимаете на пользователя, вы переходите в главный вид, который отображает список проектов. Когда вы нажимаете на проект, вы открываете доску Kanban для этого проекта. Вы увидите колонки. Вы сможете перетаскивать карточки туда и обратно между различными колонками. Вы увидите любые карточки, которые назначены вам, текущему вошедшему пользователю, другим цветом, отличным от всех остальных, чтобы вы могли быстро увидеть свои. Вы можете редактировать любые комментарии, которые вы сделали, но вы не можете редактировать комментарии, которые сделали другие люди. Вы можете удалить любые комментарии, которые вы сделали, но вы не можете удалить комментарии, которые сделал кто-либо еще.
```

### Шаг 4: Проверьте спецификацию

Проверьте чеклист спецификации с помощью команды `/speckit.checklist`:

```bash
/speckit.checklist
```

### Шаг 5: Сгенерируйте технический план с помощью `/speckit.plan`

Будьте конкретны относительно вашего технологического стека и технических требований:

```bash
/speckit.plan Мы собираемся сгенерировать это, используя .NET Aspire, используя Postgres в качестве базы данных. Фронтенд должен использовать Blazor server с досками задач с перетаскиванием, обновлениями в реальном времени. Должен быть создан REST API с API проектов, API задач и API уведомлений.
```

### Шаг 6: Проверьте и реализуйте

Попросите вашего AI-агента провести аудит плана реализации с помощью `/speckit.analyze`:

```bash
/speckit.analyze
```

Наконец, реализуйте решение:

```bash
/speckit.implement
```

## Ключевые принципы

- **Будьте явными** относительно того, что вы строите и почему
- **Не фокусируйтесь на технологическом стеке** во время фазы спецификации
- **Итерируйте и уточняйте** ваши спецификации перед реализацией
- **Проверяйте** план перед началом кодирования
- **Позвольте AI-агенту обработать** детали реализации
